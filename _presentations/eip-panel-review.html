EIP Review:
Stateless

A structured look at the proposals shaping Ethereum's state architecture.

Tree
Transition

Moving Ethereum's state from the Merkle Patricia Tree to a new structure — and how to get there.

01
Tree Transition
6800 Ethereum State Using a Unified Verkle Tree

Introduces a Verkle tree alongside the existing Merkle Patricia Tree as the first step toward fully stateless Ethereum.

  • Reduces witness sizes from ~3 kB/account (18 MB worst-case/block) to ~200 bytes/account
  • Uses Bandersnatch-curve Pedersen commitments in a unified key-value structure
  • Dual-tree approach: frozen Patricia tree + live Verkle tree during transition
02
Tree Transition
7864 Unified Binary Tree for Ethereum State

Replaces Ethereum's hexary Patricia tries with a single unified binary tree — accounts, storage, and code in one structure.

  • Designed for ZK-friendliness: enables validity proofs (SNARKs/STARKs) over blocks
  • 32-byte keys with 31-byte stem + 1-byte sub-index co-locate related data in the same branch
  • Uses BLAKE3 for hashing (final choice pending); directly addresses MPT + RLP + Keccak proof costs
03
Tree Transition
7612 Verkle State Transition via Overlay Tree

Defines the dual-tree "frozen base" approach: the existing MPT freezes and a new Verkle overlay handles all writes.

  • Reads check the overlay first, fall back to the frozen MPT — no immediate full migration needed
  • Lets the Verkle fork ship without requiring a lengthy upfront state conversion
  • Avoids consensus-layer disruptions from converting the entire historical state at once
04
Tree Transition
7748 Background State Conversion to Verkle

Defines a per-block background process that migrates MPT state into the Verkle tree, a fixed number of items at a time.

  • Converts a fixed CONVERSION_STRIDE of items before each block's transactions
  • Processes storage slots first, then account data and code
  • Organic conversion via transactions alone is too slow — dedicated migration is essential
05
Tree Transition
2926 Chunk-Based Code Merkleization

Splits contract bytecode into 31-byte chunks in a Merkle tree, so stateless clients only fetch the chunks actually accessed.

  • Bytecode is the 2nd largest contributor to witness size — this directly cuts it
  • Adds codeSize and codeRoot account fields; increases code storage gas from 200 to 500/byte
  • Requires a 7612-style transition; a smaller change than full tree restructuring
06
Tree Transition
7545 Verkle Proof Verification Precompile

Adds a precompile (address 0x21) that lets smart contracts verify stateless Ethereum proofs on-chain.

  • Supports multiple proof formats via version byte (0 = MPT, 1 = polynomial commitments)
  • Bridges and L2s can verify proofs without contract upgrades when format evolves
  • Gas cost scales dynamically with number of commitments and leaf depths
07

State Expiry
& Gas Pricing

Managing state growth and aligning gas costs with the real cost of proof generation.

08
State Expiry
7736 Leaf-Level State Expiry

Expires only leaf-level data in Verkle trees after ~6 months of inactivity, preserving structural integrity. Intended as a prompt to discuss state expiry.

  • A pragmatic alternative to earlier proposals that required address space extensions
  • Expired state can be resurrected via a new tx type with a proof of previous state
  • Reduces state bloat with minimal architectural disruption
09
State Expiry
8125 Temporary Contract Storage

New opcodes TMPLOAD/TMPSTORE for data that persists across transactions but auto-clears on a protocol-defined schedule.

  • Targets ephemeral data patterns — reduces long-term state bloat without explicit deletion
  • Two-period rollover scheme: data readable for at least one full period
  • Old period's storage cleared in constant time at period boundaries
10
Gas Pricing
8188 Tiered State Write Pricing

Introduces a tiered gas pricing model for state writes, where cost scales based on the type and impact of the write operation.

  • Differentiates gas cost between new state creation, state modification, and state expansion
  • Aims to more accurately reflect the true cost of state changes to the network
  • EIP in early draft — details subject to change
11
Gas Pricing
4762 Statelessness Gas Cost Changes

Adjusts the gas schedule to align costs with the actual expense of generating Verkle tree proofs.

  • Tracks fine-grained "access events" and "write events" — charges per tree branch traversed
  • Code deployment drops from 200 gas/byte to a flat rate; adjacent storage: 2,100 → 200 gas
  • ~200 gas per code chunk accessed during execution
12

ZK &
Execution Proofs

Zero-knowledge proofs, execution verification, and block-level optimizations.

13
ZK & Proofs
8025 Optional Execution Proofs

Lets beacon nodes validate execution payloads without running a full execution client — via cryptographic proofs shared across the consensus network.

  • Two validator modes: proof-generating nodes and stateless validators
  • Lowers hardware requirements; decouples validation cost from gas limit
  • Backward-compatible (no hard fork needed) — can be tested safely before deep integration
14
ZK & Proofs
7928 Block-Level Access Lists

Mandatory access lists appended to execution payloads, recording every account and storage location accessed during block execution.

  • Enables parallel disk I/O, parallel tx execution, and state reconstruction without re-execution
  • Size constrained to block_gas_limit / 2000 items
  • Clients verify BALs asynchronously alongside parallel operations
15
ZK & Proofs
7911 Perceptron Tree ZKP Scaling

Uses Perceptron Tree-based zero-knowledge proofs to compress multiple transactions into a unified proof by analyzing inter-transaction relationships.

  • Recursive partitioning groups similar transactions via linear perceptron classifiers
  • Combines tree commitments, classification paths, and similarity proofs into a single ZKP
  • Application-layer only — no protocol changes required
